# ADR 0020 - Connector store encryption at rest

- **Date:** 2026-07-16
- **Status:** Proposed. Adversarial review required before merge (invariant #3:
  key handling changes need an explicit adversarial-review session).
- **Deciders:** Jay, Claude Code

## Context

Jay asked the right compliance question about the hosted connector (ADR 0019):
if the connector database is stolen, subpoenaed, or leaked from a backup, what
does the other side actually hold? Until now the answer was "the plaintext of
every shared memory," because the connector store deliberately holds readable
rows so the user's AI apps can retrieve them. ADR 0019 accepted that as the
opt-in carve-out. This ADR narrows it: the AI apps still receive plaintext, but
the database at rest holds only ciphertext, and the server stores no key that
can decrypt it.

The claim this design makes true, and which the test suite makes falsifiable:
**the connector database holds only ciphertext, and the server stores no
decryption key.**

## Decision

### Envelope design

- Each account gets one random 32-byte data-encryption key (DEK), generated by
  libsodium (`randombytes_buf`). The DEK is never stored raw.
- Every shared-entry row's `{type, content}` is serialized to JSON and
  encrypted under the DEK with XChaCha20-Poly1305 AEAD
  (`crypto_aead_xchacha20poly1305_ietf_encrypt`), per-row random nonce, stored
  in the existing `content` column as `nkc1:<b64 nonce>:<b64 ct>`. The `type`
  column becomes `''`. The `nkc1:` prefix doubles as the legacy-plaintext
  detector.
- The AEAD's additional data is `"nk-conn-row-v1" || account_hash`. Binding the
  account defeats the attack that matters: transplanting one account's
  ciphertext into another account's rows. The AAD deliberately EXCLUDES
  `entry_id`, because `/client/ack` atomically remaps a row's id (server id to
  vault-local id) as pure metadata; putting the id in the AAD would force that
  path to decrypt and re-encrypt, and it must never need a key.
- The DEK is stored only WRAPPED (`crypto_secretbox_easy`, format
  `nkw1:<b64 nonce>:<b64 ct>`) under key-encryption keys (KEKs) derived from
  credentials the server sees only transiently on a request, AND a
  server-environment pepper held outside the database (below). KEK derivation is
  two-step keyed BLAKE2b: `k1 = generichash(key = credential-utf8, msg = label)`
  (the `packages/sync/src/creds.ts` pattern), then
  `KEK = generichash(key = pepper, msg = k1)`. One domain label per credential
  class: `nk-conn-kek-conn-v1` (connector token), `nk-conn-kek-pair-v1`
  (pairing code), `nk-conn-kek-code-v1` (authorization code),
  `nk-conn-kek-token-v1` (access and refresh tokens). Distinct labels mean a
  KEK from one class can never unwrap a wrap made for another; the pepper means
  no KEK is derivable from database state alone.
- All primitives are libsodium (`libsodium-wrappers`); nothing is hand-rolled
  and node:crypto contributes no cipher (invariant #3). The module is
  `apps/connector-server/src/crypto.ts`; storage never imports it (storage
  stays a dumb string store).

### The pairing-code entropy floor and the pepper (ADR 0020 crypto review)

The first design wrapped the account DEK under a KEK derived from the pairing
code alone. The pairing code is only about 40 bits (8 characters over a
32-symbol alphabet, `hash.ts`), its sha256 is stored in `pairing_codes`, and the
first design never deleted the row. A thief with the DATABASE ALONE could
therefore grind the 40-bit code offline against the stored hash in well under a
minute, derive the pairing KEK, unwrap the DEK, and decrypt every one of that
account's rows forever. That falsified the core claim. This review closes it
three ways:

- **Server-environment pepper (the real fix).** `CONNECTOR_KEK_PEPPER` is a
  32-plus-byte secret (base64) held OUTSIDE the database and mixed into EVERY
  KEK via the second BLAKE2b step above. A KEK now requires BOTH the
  request-presented credential AND the pepper, so DB state alone (credential
  hashes, wraps, ciphertext) yields no key even against the weak pairing code.
  The pepper is uniform across all four credential classes (the 256-bit token
  wraps do not strictly need it, but uniformity keeps the construction simple).
  **Fail closed:** a connector constructed over the real Neon storage with the
  pepper unset or shorter than 32 bytes THROWS at startup, so the hosted deploy
  cannot run without it. InMemory storage (local dev and tests) falls back to a
  fixed, non-secret DEV pepper; self-hosters set the same env, and running
  without it is dev-only.
- **Delete pairing codes on consume and expiry.** `consumePairingCode` now
  DELETEs the row (single atomic
  `DELETE ... WHERE code_hash = $1 AND consumed = false AND expires_at > now
  RETURNING account_hash, dek_wrap`) instead of setting `consumed = true`, and
  expired codes are refused and opportunistically GC'd. A consumed or expired
  code, and its stored wrap, simply no longer exist to grind. The pepper covers
  the only remaining window: a code still outstanding (issued, unconsumed,
  unexpired).

### Chain of custody

The server can decrypt only while a request is presenting a credential, because
every KEK is derived from a credential plaintext that is stored hash-only:

1. `/pair/start` (holds the connector token): unwrap the account wrap with the
   connector-token KEK, or on first use generate the DEK, wrap it, and write it
   race-safely (below). Re-wrap the DEK under the new pairing code's KEK and
   store that wrap on the pairing-code row.
2. `/consent` (holds the pairing code): the atomic pairing-code consume returns
   the wrap; unwrap with the pairing-code KEK; mint the authorization code and
   store the DEK wrapped under the code's KEK on the code row.
3. `/token`, code grant (holds the authorization code): the atomic code consume
   returns the wrap; unwrap with the code KEK; issue the token pair with the
   DEK wrapped separately under the new access token's KEK and the new refresh
   token's KEK (each token row carries its own wrap).
4. `/token`, refresh grant (holds the refresh token): a single atomic
   `consumeToken` (DELETE ... RETURNING) yields the row and its wrap; unwrap
   with the token KEK; mint the rotated pair with fresh wraps.
5. `/mcp` (holds the access token): `verifyAccessToken` surfaces the token
   row's wrap in `AuthInfo.extra` alongside the account hash, so the handler
   unwraps with the presented token's KEK without a second read, decrypts the
   account's rows in-process, scores them exactly as before, and returns
   plaintext to the AI app. The DEK exists only for the request.
6. `/client/entries` and `/client/pending` (hold the connector token): resolve
   the account DEK from the connector-token KEK; the push encrypts every row
   before storage, the pending read decrypts before responding. A push before
   any pairing creates the DEK through the same race-safe path. `/client/
   manifest`, `/client/ack`, and `DELETE /client/scope` are untouched: they are
   metadata-only (ids, hashes, scope labels) and never need a key.

### The four corrections from the adversarial design review (mandatory)

1. **No separate wrap table.** Each credential row (`connector_accounts`,
   `pairing_codes`, `oauth_codes`, `oauth_tokens`) gains a `dek_wrap text`
   column via idempotent single-statement ALTERs (ADR 0010 discipline). The
   wrap's lifecycle IS the credential's lifecycle: delete, expiry, and revoke
   clean up wraps automatically, with no orphan-wrap garbage collection to get
   wrong.
2. **AAD excludes entry_id.** The ack path's atomic id-remap stays
   metadata-only. Binding the account hash alone already defeats cross-account
   ciphertext transplant, which is the attack with a payoff.
3. **Atomic `consumeToken` for refresh.** A single
   `DELETE ... WHERE token_hash = $1 AND kind = 'refresh' AND expires_at > now
   RETURNING *` replaces the previous get-then-delete flow. That flow carried a
   pre-existing OAuth 2.1 rotation race, independent of encryption: two
   concurrent refreshes of the same token could both succeed. Now exactly one
   wins and the loser gets invalid_grant. The race fix and the custody hand-off
   ride the same statement.
4. **COALESCE create for the account wrap.** First-time DEK creation is a
   single `UPDATE connector_accounts SET dek_wrap = COALESCE(dek_wrap,
   $candidate) WHERE account_hash = $1 RETURNING dek_wrap`. Two concurrent
   first-writers converge: whatever the statement RETURNS is the truth, and the
   caller unwraps the returned wrap, never its own candidate.

### Client behavior on failure

If the server holds a wrap or row it cannot open for the presented credential,
content routes answer HTTP 409 `{"error": "reencrypt_required"}` (never a 500,
never partial garbage). The sync client surfaces a clear instruction: re-push
the shared scopes. A push wipes and re-creates the scopes' rows under the
current key chain; the vault is the source of truth, so nothing is lost. On
`/mcp`, a token that cannot unwrap is a 401 and the AI app re-authorizes;
an unopenable refresh token is invalid_grant and forces a full re-connect.

## Threat-model delta versus ADR 0019

Now protected (what a connector-DB-only compromise yields):

- **DB theft, leaked backup, or legal process against the database alone**
  reveals ciphertext rows, wrapped DEKs, sha256 credential hashes, and
  metadata. No stored value decrypts anything, because every KEK requires both
  a credential (stored hash-only, held live only by clients) AND the
  `CONNECTOR_KEK_PEPPER` secret, which lives in the server environment and is
  never written to the database. The low-entropy pairing code is not a way in:
  its row is deleted on consume and expiry, and while a code is outstanding the
  pepper still gates its KEK.

NOT protected (unchanged, and stated honestly):

- A **compromised server codebase or runtime** can capture credentials and
  DEKs as requests arrive and decrypt from then on. This design removes stored
  keys, not a live-server compromise.
- **Memory dumps** of a serving process can contain transient DEKs and row
  plaintext.
- **AI providers see plaintext by design**: whatever a connected app retrieves
  from a shared scope, its provider processes. Same truth as ADR 0019.

Still visible in the database (accepted metadata): scope labels, entry ids,
row counts, ciphertext lengths (which approximate content lengths), timestamps,
`entry_hash` values, account hashes, and the content-free audit ledger.

Claim wording for any public copy. The precise, DB-scoped claim: "the connector
DATABASE holds only ciphertext, and the key to read it is not in the database.
It is reconstructed per request from the app's live credential together with a
secret held in the server environment, and is exposed only under the
already-conceded runtime-compromise case." We say what we do and do not STORE,
and we MAY NOT claim "we cannot read your shared memories," because the running
server necessarily decrypts per request. "We don't store," never "we can't
read."

## Legacy plaintext is self-host-only

A row without the `nkc1:` envelope is legacy plaintext. Serving it verbatim
would let anyone who can WRITE the database inject a chosen-plaintext "memory"
that syncs into a user's vault. So passthrough is gated behind an explicit
opt-in, `NORTHKEEP_CONNECTOR_ALLOW_LEGACY_PLAINTEXT=1`, which the hosted deploy
does NOT set. Without it, a non-`nkc1:` row is SKIPPED everywhere it would be
disclosed (`/mcp` tools and `/client/pending`), never served and never synced,
content-free. Production wipes the DB (below) so there are no legacy rows anyway;
this is belt-and-suspenders, and self-hosters keeping an old DB opt in knowingly.

## Environment

- `CONNECTOR_KEK_PEPPER` (required on the hosted deploy): base64 of a 32-plus-byte
  secret held outside the database, mixed into every KEK. The server refuses to
  start on a real database without it. Rotate it only with a full DB wipe and
  re-pair, since every stored wrap is bound to the current pepper.
- `NORTHKEEP_CONNECTOR_ALLOW_LEGACY_PLAINTEXT` (self-host only, default off):
  opt in to serving pre-encryption plaintext rows.

## Dependency

`libsodium-wrappers` (pure wasm) is added to `apps/connector-server` only. It
is the audited reference JS build of the same libsodium the rest of NorthKeep
uses through sodium-native; the wasm build is required because Vercel's
serverless runtime does not load sodium-native's prebuilt addon. No network
access. (Invariant #7 note: this ADR is the required record; the dependency
ships only with Jay's OK on this ADR.)

## Migration

Rows and credentials created before this change cannot be re-wrapped: pre-0020
token, code, and pairing rows are sha256-only, so no path can ever derive their
KEKs and attach wraps. The deployment story is wipe the connector database and
re-pair (re-push then reconnect each AI app). Acceptable pre-beta: no design
partner has real data in the hosted store. The same wipe provisions the new
`CONNECTOR_KEK_PEPPER`. Legacy plaintext rows are only ever readable on a
self-host that has opted into passthrough (above).

## Verification

- `apps/connector-server/test/crypto.test.ts`: round-trips, wrong-KEK,
  wrong-account, and wrong-PEPPER failures, `parseKekPepper` validation,
  formats, legacy detection, nonce freshness.
- `apps/connector-server/test/crypto-review.test.ts`: the three review fixes.
  The production pepper guard (Neon storage with no pepper throws; too-short
  throws; InMemory runs on the DEV pepper). Delete-on-consume (a consumed
  pairing row and its wrap are gone; expired codes refused and GC'd).
  Legacy-gate (the hosted server drops a non-`nkc1:` row on `/client/pending`
  while the opt-in path serves it).
- `apps/connector-server/test/encryption.test.ts`: the full custody chain ends
  in plaintext at `/mcp` and survives refresh rotation; the concurrent
  refresh race has exactly one winner whose tokens still decrypt; the 409 and
  fail-closed 401 paths.
- `apps/connector-server/test/c3-property.test.ts`, invariant 5 (THE CANARY):
  after EVERY operation in a randomized sequence, the entire storage state is
  serialized and must not contain the seeded canary or any plaintext that ever
  passed through the server. This is the falsifiability test for the claim.
- All C1-C4 suites now seed encrypted rows through a shared test helper and
  still pass unchanged in what they prove.
